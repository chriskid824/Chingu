# 專案改造與收斂計畫 (Timeleft Pivot Plan)
> 本文件經 6 人 Agent 圓桌會議審議定稿（第一次：2026-03-20 / 第二次：2026-03-20 晚間）

---

## 🎙️ 圓桌會議紀錄

### 議題：Chingu 專案整體方向優化（功能偏離修正 + UI/UX 風格重塑）

---

### Step 1 — 議題宣讀
本次會議聚焦兩大核心問題：
1. **功能偏離**：目前專案相較於原版 Timeleft，加入了過多傳統交友軟體的功能（滑動探索、公開活動牆、開放式聊天），嚴重稀釋了「盲盒式 6 人晚餐」的核心體驗。
2. **UI/UX 風格**：介面缺乏統一的設計語言與精緻感。使用者希望將風格對齊至「Apple 健康 App」般的小清新、極簡、留白、大圓角設計。

---

### Step 2 — 🧭 Chingu PM 提案

> 我認為目前的問題可以歸納成三層：

**第一層：功能肥大**
目前 App 有 5 個底部 Tab（首頁、配對、探索、聊天、個人），但 Timeleft 的核心旅程其實只需要 **3 個入口**：
- **首頁**（報名 + 倒數 + 解鎖一站式完成）
- **聊天**（僅限晚餐後互相 Match 的人才能看到）
- **個人檔案**（個性測驗 + 飲食偏好 + 歷史紀錄）

「配對」和「探索」Tab 完全違反 Timeleft 的設計哲學，必須徹底移除。

**第二層：缺乏時間軸驅動的 UI 狀態**
首頁應該是一個「隨時間自動變化的動態儀表板」，而不是靜態的資訊列表。我建議分成 4 個時間狀態：
1. 未報名 → 2. 已報名/配對中 → 3. 週二部分解鎖 → 4. 週三完全解鎖

**第三層：UI 整體缺乏一致性**
沒有統一的設計系統（Design Token），各頁面的間距、圓角、字重都不一致。需要從零建立一套 Apple Health 風格的設計規範。

---

### Step 3 — 🛡️ Timeleft PM 審查核心價值

> 我針對 Chingu PM 的提案，有以下三點要求：

1. **「探索」和「配對」必須死透**。不是隱藏，是徹底刪除程式碼。任何讓用戶在見面前「挑選」或「瀏覽」其他人的機制，都違反我們「信任演算法、擁抱未知」的核心信條。

2. **聊天室的權限必須綁死在「雙向 👍」上**。我反對任何形式的「付費解鎖聊天」或「VIP 先聊」。這不是交友軟體，聊天室是「實體見面後的情感延伸」，絕非見面前的篩選工具。

3. **週二的「部分解鎖」資訊量要嚴格控管**。我同意顯示星座和產業 Icon，但**絕對禁止**顯示以下資訊：
   - ❌ 真實姓名
   - ❌ 照片或頭像
   - ❌ 具體年齡（只能顯示年齡段，如「25-30」）
   - ❌ 任何社群帳號連結

### Step 4 — 🎨 UI/UX 總監定義視覺規格

> Apple Health 的精髓不是「白」，而是「呼吸感」。以下是我的設計規範：

**配色系統 (Color Palette)**
| 用途 | 色碼 | 說明 |
|------|------|------|
| 主背景 | `#FFFFFF` | 純白，製造極致的呼吸空間 |
| 卡片背景 | `#F5F5F7` | Apple 標準淺灰，極淡但可區隔層次 |
| 主強調色 | `#FF6B35` → `#FF9500` | 暖橘漸層，用於倒數圓環與主 CTA 按鈕 |
| 輔助強調色 | `#34C759` | Apple 綠，用於「已確認」成功狀態 |
| 資訊色 | `#007AFF` | Apple 藍，用於連結與次要互動 |
| 文字主色 | `#1D1D1F` | 近黑色，非純黑，減少視覺壓力 |
| 文字副色 | `#8E8E93` | 中灰色，用於輔助說明文字 |

**排版規格 (Typography)**
- 字體家族：SF Pro Display（iOS 原生）/ Inter（Android fallback）
- 頁面標題：28pt, Bold (W700)
- 卡片標題：20pt, Semibold (W600)
- 內文：16pt, Regular (W400)
- 輔助文字：13pt, Regular (W400), 副色

**元件規格 (Component Specs)**
- 卡片圓角：`BorderRadius.circular(20)`
- 按鈕圓角：`BorderRadius.circular(16)`
- 卡片陰影：`BoxShadow(color: Colors.black.withOpacity(0.04), blurRadius: 20, offset: Offset(0, 4))`
- 頁面水平內距：`EdgeInsets.symmetric(horizontal: 20)`
- 卡片之間間距：`16px`
- 區塊之間間距：`32px`

**倒數圓環規格**
- 直徑：200pt
- 環寬：12pt
- 漸層方向：`SweepGradient` 從橘到黃
- 背景環：`Colors.black.withOpacity(0.05)`
- 中央文字：剩餘天數用 48pt Bold，「天」字用 16pt Regular

**動畫規格**
- 頁面切換：`CurvedAnimation` + `Curves.easeOutCubic`，時長 350ms
- 卡片出現：`SlideTransition` 從底部上移 + `FadeTransition`，時長 400ms
- 按鈕點擊：`ScaleTransition` 縮至 0.95 再回彈，時長 150ms
- 解鎖翻牌：`Transform` 3D 翻轉 180°，搭配 Haptic Feedback (medium)

### Step 5 — 💻 Tech Lead 技術可行性評估

> 以下是我的技術架構建議：

**資料結構（Firestore Collections）**
```
/DinnerEvents/{eventId}        ← 每週系統自動生成的大活動
  - eventDate: Timestamp
  - signupDeadline: Timestamp
  - status: 'open' | 'matching' | 'revealed' | 'completed'
  - signedUpUsers: string[]

/DinnerGroups/{groupId}        ← 6 人一桌的配對結果
  - eventId: string
  - participantIds: string[]
  - restaurantName / Address / Location
  - status: 'pending' | 'info_revealed' | 'location_revealed' | 'completed'
  - icebreakerQuestions: string[]

/Users/{uid}
  - dietaryPreferences: string[]  ← 新增：飲食禁忌
  - budgetRange: int
  - personalityQuiz: map          ← 個性測驗結果
```

**排程機制（Cloud Functions）**
- 每週一凌晨：自動建立下一場 `DinnerEvent`，狀態為 `open`。
- 週二中午 12:00：截止報名，觸發配對演算法，產出 `DinnerGroups`。
- 週二下午 18:00：推播「飯友部分解鎖」通知，Group 狀態改為 `info_revealed`。
- 週三下午 17:00：推播「餐廳已揭曉」通知，Group 狀態改為 `location_revealed`。
- 週四上午 10:00：推播「記得為昨晚的飯友評分」通知。

**技術風險提醒**
1. 配對演算法必須在 Cloud Function 的 60 秒 timeout 內完成，若報名人數超過 500 人需考慮分批處理。
2. 所有時間排程必須以 UTC 儲存，App 端才做時區轉換顯示。

### Step 6 — 🕵️‍♂️ 邊緣案例精算師壓力測試

> 我準備了 5 個會讓這個產品翻車的極端情境：

| # | 邊緣案例 | 嚴重度 | 建議處理方式 |
|---|---------|--------|------------|
| 1 | 報名人數不足 6 人 | 🔴 高 | 週二配對時若無法湊組，自動發放「下期免費券」並以溫暖文案通知：「為確保最佳體驗，您已被保留至下一期」 |
| 2 | 報名人數=7 人（無法整除 6） | 🟡 中 | 允許 5~7 人彈性成組。配對演算法的最終輸出不應硬性鎖死 6 人 |
| 3 | 用戶在週三下午取消（晚餐前 3 小時） | 🔴 高 | 設定「週二中午後禁止取消」的硬規則。之前可自由取消並全額退費 |
| 4 | 週三解鎖時刻高併發 | 🟡 中 | App 端採用本地快取 + 輪詢機制，Cloud Functions 分批推播（每批間隔 3 秒） |
| 5 | 晚餐後某人未完成評價 | 🟢 低 | 設定 72 小時評價期限。逾期自動視為「跳過」，不影響其他人的 Match 結果 |

### Step 7 — 🙋‍♂️ 實際使用者毒舌反饋

> 身為一個對這個 App 充滿期待但又充滿疑慮的真實使用者，我有以下問題：

1. **「報名後到週二之間，我的首頁會是什麼？」** — 如果報名後畫面就停在那裡什麼都沒有，我會覺得 App 壞掉了。請在這個階段給我一張精美的「電子邀請卡」，讓我截圖給朋友炫耀我報名了。

2. **「我不吃牛肉，整個 App 裡面要去哪裡設定？」** — 飲食偏好必須在 Onboarding 就問清楚，而且要在個人設定頁面能隨時修改。不要讓我面對一桌牛排才發現沒地方設定。

3. **「六個陌生人坐下來之後，那個『破冰話題包』一定要好玩。」** — 如果破冰問題是「你的興趣是什麼？」這種無聊的小學作文題目，我會很失望。請準備有趣、有深度、能引發爭論的問題，例如：「如果你可以和歷史上任何一個人共進晚餐，你會選誰？」

4. **「評價的時候，對方會知道我給了 👎 嗎？」** — 必須 100% 雙盲。我給了 👎 對方絕對不能知道是我。只有雙方都給 👍 才會有結果，否則就像什麼都沒發生一樣。

| 問題 | 現狀 | Timeleft 原版 | 差距 |

## 2. 產品重塑方向

收斂為一條**極簡的單一旅程**：

```
報名本週晚餐 → 等待配對 → 週二解鎖飯友輪廓 → 週三解鎖餐廳
→ 實體 6 人見面 → 隔天互評 → 雙向 👍 解鎖私訊
```

三大鐵律：
1. **🚫 拒絕事前篩選**：移除 Explore / Matching，飯友由系統全權分配。
2. **🔒 聊天必須靠實體見面解鎖**：晚餐後雙盲 👍 才開通，非交友軟體。
3. **⏱️ 時間軸驅動 UI**：首頁不是靜態頁面，是隨時間自動偵測狀態的動態儀表板。

---

## 3. 功能收斂清單

### ❌ 必須移除
| 模組 | 原因 |
|------|------|
| `lib/screens/explore/` | 破壞盲盒核心，允許事前挑人 |
| `lib/screens/matching/` | 同上，暴露用戶資料 |
| `lib/screens/events/` | 多活動模式違反「唯一一場官方晚餐」 |
| 開放式聊天入口 | 必須改為「晚餐後 Match 才解鎖」 |
| 底部導覽「配對」「探索」Tab | 精簡至 3 個 Tab |

### ✅ 必須保留並強化
| 模組 | 優化方向 |
|------|---------|
| `home/` | 改造為 4 狀態動態儀表板（報名→配對中→部分解鎖→完全解鎖）|
| `chat/` | 加入權限閘門：僅顯示已 Match 的聊天室 |
| `profile/` + `onboarding/` | 強化個性測驗 + 飲食偏好（必填）|
| `settings/` | 整合進個人檔案 Tab |

# 🎙️ 第二次圓桌會議紀錄

### 議題：UI 差異化、包容性設計、飲食選項、拉新機制、餐廳營運

---

### Step 1 — 議題宣讀

本次會議聚焦 5 個策略議題：
1. **UI 差異化**：首頁佈局不能太像 Timeleft，需要建立自己的品牌識別。
2. **LGBT 包容性**：設計上要讓所有性別取向的人都自然地使用，但不刻意強調彩虹元素，像 Tinder 那樣自然融合。
3. **素食選項**：飲食偏好系統需要更完整。
4. **拉新機制**：如何讓更多人加入平台。
5. **餐廳營運模式**：餐廳怎麼選、怎麼配、怎麼訂。

---

### Step 2 — 🧭 Chingu PM 提案

> 針對這 5 個議題，我的初步規劃如下：

**1. UI 差異化**
Timeleft 的首頁是「倒數計時器 + 報名」的直球設計。我們可以做出差異：
- 不用「冰冷的數字倒數」，改用**故事型卡片流**，像 Instagram Stories 風格地一張張往上滑。
- 首頁上半部放一句溫暖的個人化問候語（「嗨，Chris 🌿 這週要來場驚喜嗎？」），下半部才是活動卡片。
- 加入「往期回顧」區塊，讓老用戶看見自己的社交足跡（參加了幾場、認識了幾人），增加歸屬感。

**2. LGBT 包容性**
- Onboarding 的性別選項改為：男性 / 女性 / 非二元 / 不公開。不使用彩虹旗或特殊標記。
- 配對偏好改為：「你希望和誰吃飯？」→ 男性 / 女性 / 都可以 / 不在意。
- 語言和文案全面中性化，不使用「異性」「同性」等標籤。

**3. 素食**
- 飲食偏好需要擴充為多選：無限制 / 蛋奶素 / 全素(Vegan) / 不吃牛 / 不吃豬 / 海鮮過敏 / 清真(Halal) / 其他(自填)

**4. 拉新機制**
- 「帶朋友來」功能：首次參加者可以帶 1 位朋友（+1 機制），寄出專屬邀請連結，兩人會被分在同一桌。
- 「社交故事牆」：晚餐後 24 小時內，可選擇在 App 裡分享一張活動照片或一句感想（不標記其他人），形成社群內容飛輪。

**5. 餐廳**
- MVP 階段先由營運團隊手動維護一份餐廳清單，系統根據該桌 6 人的飲食偏好和預算自動配對。
- 訂位由 Cloud Function 發送通知給營運人員，由專人致電餐廳完成訂位，再回填確認資料。

### Step 3 — 🛡️ Timeleft PM 審查核心價值

> 我有幾個嚴肅的意見：

1. **「往期回顧」可以，但不能變成社群動態牆**。我同意顯示用戶自己的「參加次數」和「解鎖的朋友數」，但絕不允許做成像 Instagram 那樣的公開動態瀑布流。這不是社群軟體，是「每週一次的真實社交」工具。往期回顧只能是自己看的私人日記，不能有按讚、留言。

2. **「帶朋友來 (+1)」我有條件同意**。帶來的朋友必須一樣完成個性測驗，而且兩人雖然坐同一桌，但在 App 裡不能事先看到彼此以外其他 4 人的資訊。還有，+1 應該限制為「對方從未使用過此 App」的全新用戶，防止老用戶只跟認識的人一起報名。

3. **性別與配對偏好的設計非常好**。不標籤化、不強調，讓每個人都覺得這只是一個「認識新朋友」的場合——這才是正確的方向。我補充一點：前端 UI 上不應該在任何地方顯示其他參與者的性別取向偏好，這是純粹的後端配對參數，對用戶不可見。

### Step 5 — 💻 Tech Lead 技術可行性評估

> 以下是新增議題的技術方案：

**1. 性別與偏好的資料結構更新**
```
/Users/{uid}
  - gender: 'male' | 'female' | 'non_binary' | 'undisclosed'
  - diningPreference: 'male' | 'female' | 'any' | 'no_preference'
  - dietaryPreferences: string[]  ← 擴充為多選
```

配對演算法邏輯：
- `diningPreference: 'any'` 和 `'no_preference'` 視為等價，不做性別篩選。
- 當用戶選擇 `'male'` 或 `'female'`，演算法**盡量**但不**保證**滿足（避免人數不足時卡死配對）。
- `gender: 'undisclosed'` 的用戶在週二部分解鎖時，其性別欄位顯示為「—」。

**2. 飲食偏好的擴充**
```dart
enum DietaryPreference {
  none,           // 無限制
  lactoOvoVeg,    // 蛋奶素
  vegan,          // 全素
  noBeef,         // 不吃牛
  noPork,         // 不吃豬
  seafoodAllergy, // 海鮮過敏
  halal,          // 清真
  custom,         // 其他（自填）
}
```
- 儲存為 `string[]`，允許多選。
- 餐廳配對時取 6 人的飲食偏好**聯集**，篩選能滿足所有人的餐廳。

**3. 餐廳資料庫結構**
```
/Restaurants/{restaurantId}
  - name: string
  - address: string
  - location: GeoPoint
  - phone: string
  - budgetLevel: int (0-3)
  - maxGroupSize: int
  - dietaryTags: string[]         ← 可容納的飲食類型
  - isActive: bool
  - lastBookedAt: Timestamp       ← 防止同一家連續被選
  - city: string
  - district: string
```

**4. 訂位流程（MVP 半自動）**
```
配對完成 → Cloud Function 自動匹配餐廳
→ 產出「待訂位工單」寫入 /BookingTasks/{taskId}
→ 營運人員在後台看到工單 → 致電訂位 → 回填確認
→ 確認後觸發推播：「您的餐廳已確認！」
```

**5. +1 邀請機制**
- 邀請連結包含 `inviterUid` + `eventId`，新用戶註冊時自動帶入。
- 配對演算法將邀請者與被邀請者**綁定**分到同一桌。
- 限制：每場活動每人僅能邀請 1 位，且被邀請者必須是 App 新用戶。

### Step 7 — 🙋‍♂️ 實際使用者毒舌反饋

> 這次我有更多問題：

1. **「你說不強調 LGBT，但如果我選了『希望和男性吃飯』，然後被分到 6 個男生的桌，我會不會覺得很尷尬？」** — 配對演算法應該盡量保持每桌的多元性（性別、年齡、職業混合），而不是把同偏好的人湊在一起。「偏好」應該是軟性傾向，不是硬性篩選。

2. **「我是全素，但你們合作的餐廳會不會只有一兩道素食可選？那我整個晚上就看別人吃肉？」** — 如果桌上有全素/蛋奶素的人，應該直接指定素食友善餐廳，而不是指定一般餐廳然後期望他們有素食選項。

3. **「帶朋友來的功能很好，但我想帶的朋友已經有帳號了怎麼辦？」** — 建議增加「好友同行」功能，不限新用戶，但一個月只能使用一次，而且費用正常收取（不是免費），防止變成固定飯搭子模式。

4. **「餐廳是人工訂的話，如果訂位的人放假了呢？」** — 第二階段要做自動化訂位（串接 inline.app 或類似平台的 API），人工只做兜底。

5. **「首頁那個『你已參加 4 場、認識了 12 人』的數據卡，能不能做成像 Apple Watch 的成就徽章？」** — 很讚的建議！加入成就系統可以增加黏著度。例如：「社交新手（首次參加）」「破冰王（連續 3 場）」「美食家（嘗試 5 間不同餐廳）」。

## 12. 餐廳營運模式

### 三階段演進
```
MVP（半自動）→ 成長期（混合）→ 成熟期（全自動）
```

| 階段 | 餐廳來源 | 配對方式 | 訂位方式 |
|------|---------|---------|---------|
| **MVP** | 營運團隊手動建立餐廳清單 | 系統根據飲食偏好 + 預算自動配對 Top 3 | 系統產出工單 → 專人致電訂位 → 回填確認 |
| **成長期** | 餐廳主動申請入駐（提供折扣換曝光） | 同上 + 考量地理位置優先近多數人的地點 | 串接 inline.app / EZTABLE API 半自動 |
| **成熟期** | 開放餐廳合作後台，自助管理時段 | 全自動智能配對 | API 全自動完成預約 + 確認推播 |

### 餐廳配對演算法邏輯
```
1. 取同桌 6 人飲食偏好聯集 → 篩選合格標籤的餐廳
2. 取同桌 6 人預算範圍取交集 → 篩選價位匹配
3. 取同桌 6 人所在區域 → 優先選擇地理中心點附近
4. 排除近 2 週內被分配過的餐廳（避免重複）
5. 輸出 Top 3 候選，營運人員依序嘗試訂位
```

### 訂位失敗的備案
- 3 間候選餐廳依序嘗試（優先 → 備選 → 兜底）。
- 若全部失敗：營運人員在工單系統標記「需手動處置」，48 小時內解決。
- 極端情況（連兜底都失敗）：發放補償券 + 溫暖文案通知保留至下期。
