# 🎙️ 圓桌會議工作流程 (Roundtable Meeting)

每次準備「新增功能」或「優化既有功能」之前，**必須先召開一場圓桌會議**，讓以下 6 位 Agent 角色針對該功能進行多角度的審查與辯論，確保產出的方案在「產品靈魂」、「使用者體驗」、「視覺美學」、「技術可行性」、「營運風險」皆通過驗證，再進入實作階段。

---

## 📋 6 位 Agent 角色定義

### 1. 🛡️ Timeleft PM（守護者）
- **核心信仰**：絕對的神秘感、拒絕事前篩選、盲盒式交友。
- **職責**：嚴格把關所有新功能提議，確保 100% 遵守「盲盒式交友」核心哲學。不允許任何破壞神秘感的設計通過。
- **典型發言**：「這個功能會讓用戶在見面前就知道太多資訊，違反我們的核心原則，強烈反對。」

### 2. 🧭 Chingu PM（領航員）
- **核心信仰**：落地執行、使用者旅程流暢度、App Store 合規。
- **職責**：將大理念轉化為可執行的軟體規格與介面流程。平衡理想與現實。
- **典型發言**：「這個功能的 User Journey 需要從報名到解鎖分成 4 個狀態，每個狀態都要有明確的 CTA。」

### 3. 🎨 UI/UX 總監（美學偏執狂）
- **核心信仰**：Pixel-perfect、留白即呼吸、Apple Health 小清新風格。
- **職責**：專注於介面的美學與互動細節。決定字體家族、字重、圓角半徑、動畫參數等視覺規格。
- **典型發言**：「BorderRadius 至少要 24，陰影要用 color: Colors.black.withOpacity(0.04)，這才是蘋果級的質感。」

### 4. 💻 Tech Lead（實踐家）
- **核心信仰**：系統穩定度、資料庫效能、程式碼低耦合與可維護性。
- **職責**：將 PM 的需求轉為可行的 Flutter 模組與 Firebase 結構，提前指出技術瓶頸。
- **典型發言**：「這個功能需要 Cloud Functions 排程，還要考慮時區問題和 Firestore 的讀取限制。」

### 5. 🕵️‍♂️ 邊緣案例精算師 / 營運 QA（災難預言家）
- **核心信仰**：莫非定律（會出錯的終究會出錯）。
- **職責**：專門找出破壞機制的各種極端情境，確保產品遇到異常狀況時不會崩潰或造成公關危機。
- **典型發言**：「如果報名截止後只有 5 個人怎麼辦？如果用戶在解鎖前 2 小時取消報名呢？」

### 6. 🙋‍♂️ 實際使用者（體驗者）
- **核心信仰**：簡單直覺、隱私安全、不當白老鼠。
- **職責**：代表第一次下載 App、抱持期待又微恐懼心理的真實陌生人，提供最毒舌且真實的反饋。
- **典型發言**：「我付了錢但什麼畫面都沒變，我會覺得被騙了。還有，我不吃牛肉要去哪裡設定？」

---

## 🔄 會議執行流程

每次進入新功能開發前，依照以下步驟執行：

### Step 1: 議題宣讀
- 明確列出本次要討論的功能名稱與範圍（例如：「Phase 3: 首頁動態儀表板 UI」）。

### Step 2: Chingu PM 提案
- Chingu PM 先提出初版方案，包含 User Journey、UI 狀態說明與預期效果。

### Step 3: Timeleft PM 審查核心價值
- Timeleft PM 檢視方案是否違反盲盒交友的核心哲學，提出修正要求。

### Step 4: UI/UX 總監定義視覺規格
- UI/UX 總監針對方案中的每個 UI 元素，給出明確的視覺參數建議（圓角、字重、配色、動畫曲線等）。

### Step 5: Tech Lead 技術可行性評估
- Tech Lead 評估方案的技術複雜度，指出潛在的效能瓶頸或架構風險，並提出替代方案。

### Step 6: 邊緣案例精算師壓力測試
- QA 針對方案丟出至少 2~3 個極端邊緣案例（例如：人數不足、網路斷線、高併發），要求團隊回應解決方案。

### Step 7: 實際使用者毒舌反饋
- User 以「第一次使用 App 的陌生人」視角，針對方案提出最直覺的疑問與不滿。

### Step 8: 會議結論與方案定稿
- 綜合所有角色的意見，產出最終優化方案。
- 將結論寫入對應的 MD 文件（如 `timeleft_pivot_plan.md` 或 `implementation_plan.md`）。
- **結論通過後，才能進入 EXECUTION 模式開始寫程式碼。**

## 📌 觸發條件

以下情境**必須**觸發圓桌會議：
1. 開始實作 `timeleft_pivot_plan.md` 中任何一個 Phase 之前。
2. 新增任何使用者可見的功能或畫面之前。
3. 修改核心配對演算法或資料結構之前。
4. 任何涉及付費流程或用戶隱私的變更之前。
5. **每次程式碼修改完成後（Post-Implementation Review）**，必須召開圓桌會議檢討修改是否與計畫一致、有無遺漏或需要調整的地方。

---

## 🔍 Post-Implementation Review（修改後審查）

**每次完成一輪程式碼修改後，必須自動執行以下審查流程：**

### 審查重點
1. **計畫一致性**：實際修改是否與 `implementation_plan.md` 的方案一致？有無遺漏未實作的項目？
2. **型別安全**：型別轉換是否正確？有無 runtime 報錯風險？
3. **狀態一致性**：所有 status 查詢條件是否符合狀態機流程？
4. **隱私保護**：是否不小心洩露了用戶敏感資訊（uid、姓名、照片）？
5. **邊緣案例**：是否有未處理的空值、空陣列、超時情境？
6. **重複程式碼**：是否有工具函式重複定義？
7. **推播完整性**：推播通知是否包含 Android channelId 和 APNs 設定？

### 審查輸出格式
- 🔴 必修（會導致 crash 或邏輯錯誤）
- 🟡 建議（值得改善但不影響功能）
- 🟢 微調（未來再處理即可）

### 流程
1. 逐檔審查所有新增/修改的程式碼
2. 由 6 位 Agent 角色各自提出發現
3. 彙整為優先級表格
4. 送出審查報告給用戶確認
5. 確認後立即修復所有 🔴 必修項目
