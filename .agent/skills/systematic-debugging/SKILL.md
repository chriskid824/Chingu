# Systematic Debugging (系統化除錯)

## 概述
隨機修復會浪費時間並產生新 bug。快速補丁會掩蓋根本問題。

**核心原則:** 在嘗試修復前，必須找到根本原因。症狀修復等於失敗。

## 黃金法則
```
沒有根本原因調查，就不能提出修復方案
```

## 何時使用
適用於任何技術問題：
- 測試失敗
- 生產環境 Bug
- 異常行為
- 效能問題
- Build 失敗
- 整合問題

**特別在以下情況使用：**
- 時間壓力大時（緊急情況讓人想猜測）
- 「快速修復」看起來很明顯時
- 已經嘗試多次修復
- 上次修復沒有效果
- 不完全理解問題時

## 四個階段

### Phase 1: 根本原因調查

**在嘗試任何修復之前：**

1. **仔細閱讀錯誤訊息**
   - 不要跳過錯誤或警告
   - 它們通常包含確切的解決方案
   - 完整閱讀 stack traces
   - 記下行號、檔案路徑、錯誤代碼

2. **穩定重現問題**
   - 能夠可靠地觸發嗎？
   - 確切的步驟是什麼？
   - 每次都會發生嗎？
   - 無法重現 → 收集更多數據，不要猜測

3. **檢查最近的變更**
   - 什麼改變可能導致這個問題？
   - Git diff、最近的 commits
   - 新的依賴、配置變更
   - 環境差異

4. **在多組件系統中收集證據**
   - 在每個組件邊界記錄進入和離開的數據
   - 驗證環境/配置的傳播
   - 檢查每一層的狀態


### Phase 2: 模式分析

1. **找到可運作的範例**
   - 在同一程式碼庫中找到類似的可運作程式碼
   - 什麼是類似但可運作的？

2. **與參考進行比較**
   - 如果實現模式，完整閱讀參考實現
   - 不要略讀 - 閱讀每一行

3. **識別差異**
   - 可運作和損壞之間有什麼不同？
   - 列出每個差異，無論多小

### Phase 3: 假設和測試

1. **形成單一假設**
   - 清楚陳述：「我認為 X 是根本原因，因為 Y」
   - 寫下來，具體而非模糊

2. **最小化測試**
   - 進行最小可能的變更來測試假設
   - 一次只改變一個變數
   - 不要同時修復多個問題

3. **驗證後再繼續**
   - 有效？→ Phase 4
   - 無效？→ 形成新假設
   - 不要在上面添加更多修復

### Phase 4: 實施

1. **創建失敗的測試案例**
   - 最簡單的重現
   - 如果可能的話使用自動化測試
   - 修復前必須有

2. **實施單一修復**
   - 解決已識別的根本原因
   - 一次一個變更
   - 沒有「順便」的改進

3. **驗證修復**
   - 測試現在通過了嗎？
   - 沒有其他測試壞掉？
   - 問題真的解決了？

4. **如果 3+ 次修復失敗**
   - 停下來質疑架構
   - 這個模式根本上是否健全？
   - 應該重構架構而非繼續修復症狀？

## 危險信號 - 停下來遵循流程

如果你發現自己在想：
- 「先快速修復，稍後調查」
- 「試著改變 X 看看是否有效」
- 「添加多個變更，運行測試」
- 「我沒有完全理解，但這可能有效」

**以上都意味著：停下來。返回 Phase 1。**

## 快速參考

| 階段 | 關鍵活動 | 成功標準 |
|------|---------|---------|
| 1. 根本原因 | 閱讀錯誤、重現、檢查變更 | 理解什麼和為什麼 |
| 2. 模式 | 找到可運作的範例、比較 | 識別差異 |
| 3. 假設 | 形成理論、最小化測試 | 確認或新假設 |
| 4. 實施 | 創建測試、修復、驗證 | Bug 解決、測試通過 |

## 實際影響

從除錯會話的數據：
- 系統化方法：15-30 分鐘修復
- 隨機修復方法：2-3 小時的掙扎
- 首次修復成功率：95% vs 40%
- 引入新 bug：幾乎為零 vs 常見
