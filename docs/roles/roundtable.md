# 圓桌會議機制

## 定位
圓桌會議是 Chingu 專案的多角色審查機制。讓 8 個角色針對同一議題，從各自專業角度發言、提出問題、發現衝突，最終整合成明確的行動方案。

---

## 觸發方式

### 手動觸發
用戶說出以下任一指令即啟動：
- 「開圓桌」「開個圓桌」「圓桌會議」
- 「針對 XXX 開圓桌」
- 「讓大家討論一下 XXX」

### 指定角色
- 「讓 PM 和 Tech Lead 討論 XXX」→ 只有指定角色發言
- 「用 PM 的角度看一下」→ 單一角色快速審查

### 主動建議觸發（Claude 應主動提醒）
當偵測到以下情境時，Claude 應主動詢問「要不要開個圓桌討論一下？」：

| 情境 | 說明 |
|------|------|
| 新增功能 | 用戶提出規格書中沒有的新功能 |
| 修改核心流程 | 報名、配對、解鎖、評價、聊天的邏輯變動 |
| 資料結構大改 | 新增/刪除 Collection，或修改核心欄位 |
| UI 大改 | 首頁、底部導覽、Onboarding 的結構性變動 |
| 營運流程變動 | 時間軸調整、成團規則改變、餐廳指定方式變更 |
| 上線/發版前 | 即將部署到正式環境 |
| 方向衝突 | 用戶的需求可能與產品鐵律衝突 |
| 技術方案有多個選項 | 需要權衡取捨的架構決策 |

### 不需要圓桌的情況
- 修一個明確的 Bug（直接修）
- 調整文案措辭（直接改）
- 套用已定義的 Design Token（直接做）
- 寫測試（直接寫）
- 規格已明確、無歧義的實作（直接做）

---

## 會議流程

### 完整圓桌（7 步，全員 8 角色）

```
Step 1 — 議題宣讀
  Claude 簡述議題背景和需要決策的問題

Step 2 — Timeleft PM 發言
  - 是否符合產品核心信條？
  - 在 Timeleft 原版中有對應嗎？
  - 是否偏離盲盒精神？
  - 結論：支持/反對/有條件支持

Step 3 — UI/UX 總監發言
  - 視覺和體驗怎麼設計？
  - 是否符合 Design Token？
  - 工作量評估
  - 結論：設計建議或反對理由

Step 4 — Tech Lead 發言
  - 技術上可行嗎？
  - 架構影響範圍？
  - 有沒有更簡單的做法？
  - 結論：技術方案建議

Step 5 — 專業角色發言（按議題相關性選擇）
  - DBA：資料結構影響？需要新 Collection/欄位嗎？
  - DevOps：CI/CD 影響？需要新的部署流程嗎？
  - 營運管理員：營運流程影響？需要新的 SOP 嗎？
  - QA 工程師：測試影響？需要多少新測試？

Step 6 — User 測試員發言
  - 用戶會怎麼想？
  - 有什麼體驗上的坑？
  - 文案語氣合適嗎？

Step 7 — 結論整合
  - 統整所有角色意見
  - 列出共識和分歧
  - 輸出明確的行動方案（做/不做/怎麼做）
  - 如果決定做：列出需要修改的文件和規格
```

### 精簡圓桌（3~4 角色）

適用於範圍較小的決策，只召集相關角色。例如：
- UI 問題 → PM + UI/UX + User 測試員
- 資料結構問題 → Tech Lead + DBA + QA
- 營運問題 → PM + 營運管理員 + Tech Lead
- 部署問題 → DevOps + Tech Lead + QA

流程同上，但只有被召集的角色發言。

### 單一角色快速審查

適用於需要特定角度的意見，不需要完整會議。
- 「用 PM 角度看一下」→ 只有 PM 的審查清單跑一遍
- 「DBA 看一下這個結構」→ 只做資料結構審查

---

## 發言規則

1. **每個角色只代表自己的職責**，不越權評論其他領域
2. **必須給出明確結論**：支持/反對/有條件支持，不能說「都可以」
3. **反對時必須說理由**，不能只說「我覺得不好」
4. **意見衝突時，由用戶做最終決策**
5. **發言要簡潔**，每個角色 3~5 句話，不寫長篇大論

---

## 產出格式

圓桌結束後，輸出結論摘要：

```
## 圓桌結論：{議題名稱}

### 決議
{做/不做/延後/修改後再議}

### 理由
- {核心理由 1}
- {核心理由 2}

### 行動項目（如果決定做）
- [ ] {具體行動 1}
- [ ] {具體行動 2}

### 需更新的規格文件
- {檔案 1}：{更新內容摘要}
- {檔案 2}：{更新內容摘要}
```

---

## 規格同步

圓桌會議做出的決策，必須立即同步到對應的規格文件：
- 產品決策 → 更新 `CLAUDE.md` 和 `timeleft_pm.md`
- UI 決策 → 更新 `uiux_director.md`
- 資料結構決策 → 更新 `dba.md`
- 營運決策 → 更新 `ops_manager.md`
- 架構決策 → 更新 `tech_lead.md`
- 部署決策 → 更新 `devops.md`
- 測試決策 → 更新 `qa_engineer.md`

**不可以只在對話中決定而不更新文件**，否則下次對話會讀到過時的規格。
