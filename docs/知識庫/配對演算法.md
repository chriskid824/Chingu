# 配對演算法(人的分桌 + 餐廳指派)

**週二 12:00 報名截止後由 Cloud Function 執行;用戶端零參與、零可見。**

## 分桌優先級(MVP)

1. **城市/地區** — 硬性,必須同區域(MVP 只有信義區)
2. **用餐偏好(想認識的性別)** — 硬性優先:男→男/女→女盡量湊同偏好一桌,湊不滿不硬湊、保留到下週;其餘(男→女、女→男、都可以、隨緣)混合分組、盡量性別平衡
3. **預算範圍** — 硬性,交集必須存在
4. **飲食偏好** — 盡量避免衝突
5. **年齡段** — 軟性,盡量同齡
6. **興趣** — 軟性,有交集加分

成團人數:**彈性 5~7 人**,不足 5 人本週取消保留至下週。

## 行為原則

- 偏好為**軟性約束**:盡量滿足但不保證,避免人數不足時卡死
- 每桌追求「多元混合」(性別、年齡段、職業盡量分散),非同類聚集
- `gender`、`diningPreference` 前端永不顯示給其他用戶

## 餐廳配對邏輯

```
1. 同桌飲食偏好聯集 → 篩選合格標籤餐廳
2. 同桌預算範圍交集 → 篩選價位
3. 同桌所在區域 → 優先地理中心點附近
4. 排除近 2 週被分配過的餐廳
5. 輸出 Top 3 候選,營運人員依序訂位
```

程式錨點:`functions/src/restaurantAssignment.ts`(餐廳指派)、`bookWithValidation.ts`(訂位驗證)。餐廳資料在 Firestore `restaurants`(名稱、地址、預算等級、飲食標籤),由後台/開發者維護。

## 技術約束

- 配對必須在 Cloud Function 60 秒 timeout 內完成;報名 >500 人需分批
- 輸出寫入 `dinner_groups`(participantIds、restaurant 資訊、icebreakerQuestions)
