# 開發慣例與流程

**寫程式前先知道:八角色審查、圓桌會議、開發規則、CI/CD、測試工具。**

## 八角色審查制度

功能開發前依情境套用視角,詳細定義在 `docs/roles/`:

| 情境 | 角色 | 文件 |
|---|---|---|
| 功能規劃 | Timeleft PM(守鐵律)+ 營運管理員 | timeleft_pm.md / ops_manager.md |
| UI 設計 | UI/UX 總監(Design Token 合規) | uiux_director.md |
| 資料結構變動 | DBA(Collection/Rules/索引) | dba.md |
| 寫程式 | Tech Lead(Provider 架構) | tech_lead.md |
| 測試 | QA 工程師 | qa_engineer.md |
| 部署 | DevOps(CI/CD、簽章、版本) | devops.md |
| 驗收 | User 測試員(體驗、文案語氣) | user_tester.md |

## 圓桌會議(roundtable.md)

需要開:規格外新功能、改核心流程(報名/配對/解鎖/評價/聊天)、增刪 Collection、結構性 UI 變動、營運時間軸調整、上線前、與鐵律衝突、技術方案權衡。
不需要:修 bug、調文案、套 Token、寫測試。
**圓桌決策必須同步回規格文件**,不可只留在對話。歷史決策紀錄在 `timeleft_pivot_plan.md`(兩次會議)。

## 開發規則(CLAUDE.md)

1. 改完必 Hot Reload 驗證;2. UI 變動前後截圖對比;3. 認證變動測完整登入流程;4. 同步更新 CLAUDE.md;5. **Commit message 用中文**。
文案一律繁體中文。色彩一律 `AppColorsMinimal`。

## CI/CD 與發佈

- GitHub Actions → Firebase App Distribution(`.github/`)
- Release Keystore 簽章已設;App Store 上架文件:`APP_STORE_METADATA.md`、`APP_STORE_MANUAL_STEPS.md`
- 強制更新走 Firebase Remote Config(`force_update_service.dart`)

## 測試工具

- 自動化:`flutter test`(13 檔,清單見 `TESTING_CHECKLIST.md`);手動:`MANUAL_TESTING_CHECKLIST.md`
- **Seeder**:debug_screen 內建測試情境產生器(6 模組 15 個 User Case),寫入權限綁 `isDeveloper()`(chriskid824@gmail.com)。改 firestore.rules 前先想 Seeder 會不會壞(踩過多次 permission-denied)

## 本地知識工具

- **Wiki**:`docs/知識庫/`(本目錄),入口 [INDEX.md](INDEX.md)
- **RAG 檢索**:`python3 tools/rag/rag.py index` 重建索引、`python3 tools/rag/rag.py search "關鍵字"` 查詢(sqlite FTS5,中文可用,涵蓋 md+dart+ts)。改完大量文件或程式後記得重建索引。
