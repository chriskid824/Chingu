# 雙盲評價機制

**晚餐後 72 小時內 👍/👎 互評;雙向 👍 即時解鎖一對一聊天。雙盲是資料庫層保證。**

## 規則

- 週五 10:00 推播提醒;顯示同桌飯友(用戶設定的名字 + 幾何頭像/照片)
- 逐一評價:只有 👍(想再聯絡)/ 👎(就到這吧)——無星級、無文字
- **雙盲**:對方 100% 不知道你給了什麼;只有雙方都 👍 才有結果,否則如同沒發生
- **即時結算**:A 給 B 👍 且 B 已給 A 👍 → 立刻建立一對一聊天室 + 推播「恭喜!你們 Match 了」
- 72 小時未評價 → 自動視為全部「跳過」(不阻塞別人的 Match);逾期處理在 `functions/src/scheduledNotifications.ts`
- 「認識了 Y 人」統計 = 所有同桌過的人(不限 Match)
- 往期回顧**不顯示**你對誰給了什麼(雙盲原則延伸)

## 雙盲的資料庫層實作

`dinner_reviews` 的 rules:`allow read: if resource.data.reviewerId == request.auth.uid` — 只能讀自己寫的評價,**查詢別人給你的評價在 rules 層就被擋掉**。評價不可修改;create 已加驗證(reviewerId 必須是本人、result 限 like/dislike),防偽造雙向 like 強制配對。

## Match 結算真相(2026-07-11 查證)

**唯一結算點是 `functions/src/pushNotifications.ts` 的 `onMutualMatch`** — `dinner_reviews` 的 onCreate trigger,用 Admin SDK(不受 rules 限制)反向查詢對方是否也給了 like。成立時:
1. 用確定性 id `match_{uidA}_{uidB}_{eventId}` + **`create()`**(非 get+set)建一對一聊天室 — 兩人同時送出評價時只有一邊成功,推播/系統訊息/totalMatches 不會重複
2. 寫系統訊息、推播雙方「配對成功」、雙方 totalMatches +1

**client 端不做結算**:review_service 曾有一套 client 結算(查對方 review + 建房),因雙盲 rules 必吃 permission-denied,是害提交流程報錯的死碼,2026-07-11 已刪除。UI 得知配對成功的管道 = 推播 + 聊天列表刷新。**不要在 client 加回反向查詢**。

評價 doc id 也是確定性的:`{reviewerId}_{revieweeId}_{groupId}`(client 提交與 autoSkipReviews 同一套),重複提交/重跑天然冪等。
