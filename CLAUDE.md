# Chingu — 盲盒式 6 人晚餐社交 App

## 專案定位
對標 Timeleft，核心體驗是「每週一場官方晚餐，5~7 個陌生人一桌，見面前零資訊」。
這不是交友軟體，是「信任演算法、擁抱未知」的真實社交工具。
與 Timeleft 錯峰：Timeleft 週三晚餐，Chingu **週四晚餐**。

## 目標市場
MVP 僅限**台北市信義區**，後期再擴展。

## 語言
所有溝通與文案使用**繁體中文**。

### 審查流程
1. **功能規劃時** → Timeleft PM 檢查是否偏離核心 + 營運管理員確認營運可行性
2. **UI 設計時** → UI/UX 總監檢查 Design Token 合規
3. **資料結構變動時** → DBA 檢查 Collection 結構與 Security Rules
4. **寫程式時** → Tech Lead 檢查架構與程式碼品質
5. **測試時** → QA 工程師撰寫/執行自動化測試
6. **部署時** → DevOps 檢查 CI/CD 和 Release checklist
7. **驗收時** → User 測試員做體驗回饋

### 圓桌會議機制
詳細流程見 `docs/roles/roundtable.md`。

**觸發方式**：用戶說「開圓桌」「圓桌會議」「針對 XXX 開圓桌」即啟動。

**主動建議觸發**：當偵測到以下情境，Claude 應主動詢問「要不要開個圓桌討論一下？」：
- 新增規格書中沒有的功能
- 修改核心流程（報名、配對、解鎖、評價、聊天）
- 新增/刪除 Firestore Collection 或修改核心欄位
- 首頁、導覽、Onboarding 的結構性 UI 變動
- 營運流程或時間軸調整
- 上線/發版前
- 用戶需求可能與產品鐵律衝突
- 技術方案有多個選項需權衡

**不需要圓桌**：修 Bug、調文案、套 Design Token、寫測試等已有明確規格的工作。

**圓桌決策必須同步更新到對應的規格文件**，不可只在對話中決定。

## 產品三大鐵律（不可違反）

1. **拒絕事前篩選**：無 Explore、無 Matching、無配對卡片。飯友由系統 100% 分配。
2. **聊天靠實體解鎖**：一對一私訊需晚餐後雙盲互評，雙向 👍 才開通。群組聊天於完全解鎖時（週三 17:00）自動建立。
3. **時間軸驅動 UI**：首頁是隨時間自動變化的動態儀表板，不是靜態頁面。

## MVP 範圍

### MVP 包含
- 註冊/登入（Email + Google + Apple）
- Onboarding（基本資料 + 照片選填 + 飲食偏好 + 預算 + 用餐偏好）
- 報名本週晚餐（僅台北市信義區）
- 首頁 4 狀態動態儀表板
- 配對演算法（Cloud Function）
- 階段式解鎖（週二飯友輪廓 → 週三餐廳地址 → 週四 19:00 照片解鎖）
- 破冰話題包（Firestore 動態載入，分 3 層級各 3~4 題）
- 晚餐後雙盲 👍/👎 評價（用戶設定的名字辨識，即時結算）
- 雙向 👍 解鎖一對一聊天室
- 群組聊天（完全解鎖時自動建立，永久保留）
- 往期回顧（Events Tab — 歷史晚餐紀錄）
- 個人檔案 + 設定
- 4 Tab 導覽
- 強制更新機制（Firebase Remote Config）
- 營運後台網站（Flutter Web）
- 推播通知（完整時間軸通知）

### MVP 不含（之後再做）
- 付費機制 / Paywall（MVP 免費）
- 個性測驗（MVP 用興趣 + 基本資料做簡易配對）
- +1 邀請機制
- 成就徽章系統

## 使用者核心旅程

```
報名本週晚餐 → 等待配對（電子邀請卡）
→ 週二 18:00 解鎖飯友輪廓（星座/產業/年齡段，抽象幾何頭像）
→ 週三 17:00 解鎖餐廳地址 + 群組聊天開通
→ 週四晚間：5~7 人實體見面 + 破冰話題包
→ 週四 19:00：群組聊天內照片全部解鎖
→ 週五起 72 小時：雙盲 👍/👎 互評
→ 雙向 👍：解鎖一對一私訊聊天室
```

## 首頁 4 狀態（動態儀表板）

| 狀態 | 觸發條件 | UI |
|------|---------|-----|
| 未報名 | 用戶未報名本週 | 倒數圓環「距截止 X 天」+ CTA「立刻報名」 |
| 配對中 | 已報名，待配對 | 電子邀請卡 + 脈衝動畫「最佳飯友匹配中...」 |
| 部分解鎖 | 週二 18:00 後 | 邀請卡翻轉，飯友模糊卡片（星座+產業+年齡段） |
| 完全解鎖 | 週三 17:00 後 | 餐廳地圖卡片 + 導航按鈕 + 破冰話題包入口 |

## 底部導覽（4 Tab）

1. **首頁** — 報名 + 倒數 + 4 狀態儀表板 + 個人統計小卡（已參加 X 場 / 認識了 Y 人）
2. **聊天** — 群組聊天（完全解鎖後建立）+ 一對一私訊（雙向 👍 後建立）
3. **Events** — 我的歷史晚餐紀錄（往期回顧）
4. **我的** — 個人檔案 + 設定

## 配對演算法（MVP 優先級）

1. **城市/地區** — 硬性，必須同區域（MVP 只有信義區）
2. **用餐偏好（想認識的性別）** — 硬性優先
   - 男→男：盡量湊一桌同偏好的男性
   - 女→女：盡量湊一桌同偏好的女性
   - 湊不滿不硬湊，保留到下週
   - 其餘（男→女、女→男、都可以、隨緣）：混合分組，盡量性別平衡
3. **預算範圍** — 硬性，交集必須存在
4. **飲食偏好** — 盡量避免衝突
5. **年齡段** — 軟性，盡量同齡聚在一起
6. **興趣** — 軟性，有交集加分

成團人數：**彈性 5~7 人**。不足 5 人本週取消，保留至下週。

## 照片顯示規則

- Onboarding 可上傳照片（選填），未上傳者使用系統分配的抽象幾何頭像
- 見面前一律使用抽象幾何頭像（部分解鎖也不顯示照片）
- **週四 19:00**（晚餐當天）：群組聊天室內所有人照片解鎖
- 一對一聊天室（Match 後建立）：直接顯示真實照片

## 評價機制

- 週五 10:00 推播提醒評價
- 顯示同桌飯友（用戶設定的名字 + 幾何頭像/照片）
- 逐一評價：只有 👍（想再聯絡）/ 👎（就到這吧），無星級、無文字
- 雙盲：對方 100% 不知道你給了什麼
- 即時結算：A 給 B 👍 且 B 已給 A 👍 → 立刻建立一對一聊天室 + 推播
- 72 小時未評價 → 自動視為全部「跳過」
- 「認識了 Y 人」= 所有同桌過的人（不限 Match）

## 破冰話題包

- 來源：Firestore `/IcebreakerQuestions` 動態載入
- 每場隨機分配 10 題，分 3 層級：暖場（前 3 題）→ 深入（中 4 題）→ 靈魂拷問（後 3 題）
- UI：卡片式，一次一題，左右滑動切換，字要大（餐桌上使用）
- 週三完全解鎖後出現入口
- 晚餐隔天可在往期回顧中回看

---

## 往期回顧（Events Tab）

每場歷史晚餐卡片顯示：
- 日期（如「2026/03/27 週四」）
- 餐廳名稱
- 同桌人數（如「6 人」）
- Match 到幾位（如「認識了 2 位新朋友」）
- 可點進看：同桌飯友列表 + 當晚破冰話題回顧
- 不顯示：你對誰給了 👍/👎（雙盲原則）
